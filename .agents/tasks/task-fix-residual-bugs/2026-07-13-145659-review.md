# Expanded series classification and manufacturer filter for combo builder (v2)

This change adds `UX_EXPANDED` and `BX_EXPANDED` enum values to the Prisma schema, reclassifies blades into the correct expanded series, fixes seed data naming inconsistencies (Whale Wave, Discball), removes orphaned entries, and adds a manufacturer filter toggle to the combo builder form. The prior review's blocking concern — the missing `bit-disk-ball` slug in orphan cleanup — has been addressed.

**Watch for:** The suppressed `exhaustive-deps` lint rule in the phantom-selection `useEffect` is an accepted tradeoff (confirmed, minor). No blocking concerns remain.

**Verdict**: APPROVED

## High-level view

The schema and migration correctly add both expanded enum values. The seed data layer creates `uxExpandedBladeNames` and `bxExpandedBladeNames` sets that slot into the fallback chain before CX and BX defaults, so Bullet Griffon, Glory Valk, and Dran Strike all resolve to the correct series without explicit `series` fields on every wiki entry.

The orphan cleanup now targets all three stale slugs (`whales-wave`, `bit-disc-ball`, `bit-disk-ball`) before the upsert loop, covering both the old metaParts slug and the old wiki-generated slug.

The combo form's manufacturer filter applies uniformly across all three part categories (blades, ratchets, bits) and the phantom-selection effect correctly clears any selection whose ID falls outside the newly-filtered set. The `bxUxBlades` filter was widened to include `UX_EXPANDED` and `BX_EXPANDED`, making expanded-line blades visible in the BX/UX tab.

## Phantom-selection clearing and the exhaustive-deps suppression

The manufacturer filter state (`"ALL" | "TAKARA_TOMY" | "HASBRO"`) drives a `filterByManufacturer` helper that narrows the `blades`, `ratchets`, and `bits` arrays before they reach the dropdown rendering. The phantom-selection `useEffect` rebuilds ID sets from the filtered arrays on every `manufacturerFilter` change and clears any selection not present. The eslint-disable for `react-hooks/exhaustive-deps` is deliberate: including the eight selection variables would cause the effect to re-run on every keystroke, defeating its purpose. The pattern works correctly under React's synchronous batching guarantees.
