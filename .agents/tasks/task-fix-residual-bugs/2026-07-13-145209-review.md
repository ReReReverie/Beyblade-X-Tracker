# Expanded series classification and manufacturer filter for combo builder

This change fixes four residual bugs spanning seed data integrity and UI behavior. It adds `UX_EXPANDED` and `BX_EXPANDED` enum values to the Prisma schema, reclassifies blades into the correct series (Bullet Griffon, Glory Valk, Dran Strike), fixes naming inconsistencies (Whale Wave, Discball), adds a manufacturer filter toggle to the combo form, removes brand suffixes from dropdown labels, and cleans up orphaned database rows from prior incorrect seeds.

**Watch for:** The orphaned-row cleanup in `seed-catalog.ts` misses `bit-disk-ball` — the slug produced by the *old* wiki entry "Disk Ball" — and only deletes `bit-disc-ball` (the old metaParts slug) (confirmed). The `filterByManufacturer` closure inside the phantom-selection `useEffect` is technically correct but relies on an eslint-disable for exhaustive-deps, making future refactors fragile (confirmed, minor).

## High-level view

The schema migration adds two new `PartSeries` enum values (`UX_EXPANDED`, `BX_EXPANDED`) and the seed data layer reclassifies three blades into them. The `applyMetaPartMetadata` fallback chain now checks the new sets before falling back to BX, which means any blade added to `uxExpandedBladeNames` or `bxExpandedBladeNames` gets the correct series without needing an explicit `series` field in its metaPart entry.

The combo form gains a manufacturer filter that sits below the series tabs. The filter narrows all three part dropdowns (blades, ratchets, bits) and a `useEffect` clears any selections that become invisible after a filter change, preventing phantom form submissions. The `bxUxBlades` filter was widened to include `UX_EXPANDED` and `BX_EXPANDED` so expanded-line blades appear in the BX/UX tab.

Seed cleanup deletes two known orphan slugs before the upsert loop, but a third orphan (`bit-disk-ball`, the wiki-generated slug from `"Disk Ball"`) is not targeted and will persist in any database that was seeded before this fix.

## Orphaned slug cleanup misses `bit-disk-ball`

The seed script now runs `prisma.partCatalog.deleteMany` targeting slugs `["whales-wave", "bit-disc-ball"]` before upserting. The `bit-disc-ball` slug came from the explicit `metaParts` entry (name "Disc Ball") which is removed in this diff. But the *wiki-generated* entry used `"Disk Ball"` (with a 'k'), which `slugifyWikiName` turned into `bit-disk-ball`. After this change, `wikiBitNames` has `"Discball"` → slug `bit-discball`. The old `bit-disk-ball` row will never be upserted again and is not in the delete list, leaving it orphaned.

```ts
// seed-catalog.ts — current cleanup
await prisma.partCatalog.deleteMany({
  where: { slug: { in: ["whales-wave", "bit-disc-ball"] } }
});
// Missing: "bit-disk-ball"
```

## Phantom-selection clearing relies on suppressed deps

The phantom-selection `useEffect` builds ID sets from the newly-filtered arrays and clears any selection not present. The eslint-disable suppresses warnings about the six selection state variables not being in the dependency array. This is the intended pattern (only fire on filter change, read current selections at that moment), but it means the effect silently relies on React's synchronous state batching to see up-to-date selection values. If any of these selections were ever moved to `useReducer` or derived state, the effect would need revisiting. A handler-based approach (clearing inside the `setManufacturerFilter` onClick) would make the data flow explicit.
